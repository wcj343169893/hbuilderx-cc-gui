### 执行前校验
如果前文没有提到需要实现的平台（Android、iOS、鸿蒙等），必须先向用户确认，**禁止**进入执行逻辑步骤。

### 执行逻辑

- Step 1. **准备工作**: 
  - 调用 skill `uniappx-pitfalls-and-tips` 获取 UTS 插件开发指南，重点了解 UTS 插件开发规范，包括接口设计、原生混编目录结构、原生混编代码注意事项、前端调用注意事项以及需要使用哪些原生 API 等。在获得结果前，**禁止**进入下一步。
  - 判断获取到的信息是否足够支撑你完成插件的实现，是则直接进入 Step 2，否则继续学习文档 `{knowledges_base_dir}/uni-app-x/docs/plugin/uts-plugin.md`，了解 uts plugin 的目录规范，学习文档 `{knowledges_base_dir}/uni-app-x/docs/plugin/uts-plugin-hybrid.md`，了解混编相关规范。
  - 生成插件目录结构以及创建对应文件。
  **注意** 
  - 必须引用文档原文，禁止在缺少依据时仅凭经验实现。

- Step 2. **定义接口**: 根据用户需求，定义插件的API接口到文件 `/uni_modules/<module_name>/utssdk/interface.uts`，以及统一的错误码定义文件 `/uni_modules/<module_name>/utssdk/unierror.uts`。
  **注意** 
  - `interface.uts` 导出 API 时不要写 `export default`，**禁止**生成 `inteface Uni {}`。
  **执行要求**
  - 在本步骤完成前，**严禁**进入 Step 3 及后续任何步骤。
  - `interface.uts` 和 `unierror.uts` 完成后，必须立即执行一次**全平台**的静态代码扫描（`lsp lint`）；只有在确认不存在明确语法错误，或已修复完明确语法错误后，才允许进入 Step 3。

- Step 3. **确定实现方案**: 涉及调用原生平台能力的情况必须使用原生混编方案，各平台的原生代码将在 Step 4 中实现，**禁止**自行决定使用 UTS 直调原生 API 或 SDK。

- Step 4. **各平台原生代码实现**: 编写各平台原生的代码，如 Android 平台的 kotlin 代码、iOS 平台的 Swift 代码、鸿蒙平台的 ets 代码、web 和小程序平台的 js 代码。
  **必须委派** **必须**将此类任务委派给子智能体**uts_native Subagent**处理。
  **必须并行** 多平台插件开发时，委派给对应数量的子智能体**并行**实现。即 Android、iOS、鸿蒙等平台的代码编写为并行方式，互不等待。
  **职责限制** 禁止自行生成原生代码，以避免不必要的上下文干扰。子智能体**禁止**生成平台胶水层代码。
  **提供上下文** 委派时，需明确要求各个子智能体学习文档 `{knowledges_base_dir}/uni-app-x/docs/plugin/uts-plugin-hybrid.md`，了解混编相关规范并严格按照 `interface.uts` 中定义的接口实现原生代码。子智能体**禁止**加载 skill、执行代码搜索、文本搜索、网络搜索等能力；只有在确实无法完成实现或确有必要补充参考时，才允许搜索。

  **执行要求**
  - Android 原生混编 `kotlin/java` 的包名必须统一使用 `uts.sdk.modules.<插件名称小驼峰>`。例如插件目录名/模块名为 `get-battery-info`，则包名必须写为 `uts.sdk.modules.getBatteryInfo`；`index.uts` 中的导入路径也必须与该包名完全一致。禁止继续使用其他自拟包名格式。
  
- Step 5. **平台胶水层代码实现**: 在各自平台目录下创建 `uni_modules/<module_name>/utssdk/<platform_name>/index.uts` 实现 Step 1 中的 `interface.uts` 定义的接口以及调用 Step 4 中生成的原生混编代码（如有）。
  **注意**
  - 如果确实需要偏离官方示例或文档写法，必须先在输出中明确说明“为什么官方示例不适用”，并引用对应文档依据；不能凭经验直接改写。
  - **禁止**在 `utssdk` 实现目录创建 `.d.ts` 文件。

- Step 6. **静态代码扫描（首轮）**: 完成代码后，必须对本次新增或修改的每一个 `.uts`、`.uvue` 文件先执行一轮静态语法检查，直接使用 skill `uniappx-syntax-checker`，并**必须并行**执行。
  **执行要求**:
  - 必须逐文件检查，不能只检查插件实现而漏掉示例页面、插件入口、接口声明、错误定义文件。
  - 首轮 `lsp lint` 的目标是“快速分流”，而不是“一次性清零所有告警”；禁止长时间围绕同一类 `lsp` 导包 / 原生符号问题反复修改代码。
  - 对“非常明确的语法错误”必须优先修复并重跑对应文件，例如：UTS 语法不合法、条件表达式不是 `boolean`、明显写错 API 名、对象字面量/类型声明写法不符合文档、模板缺少必要结构等。
  - 对“非明确语法错误”的诊断，尤其是导包问题、模块找不到、混编原生符号找不到、IDE 类型声明不一致等，允许先记录并继续进入 Step 7 做真实编译交叉验证；不能仅凭这类 `lsp` 诊断就改实现方案。
  - 如果输出的是 `Unexpected error`、`TypeScript Server Error`、`Cannot read properties of null` 等 HBuilderX / tsserver 内部异常，必须先判定为工具问题，不能直接当成代码错误处理。

  **注意**
  - 条件语句和三元表达式的条件必须是严格 `boolean`，不能依赖 truthy / falsy。
  - `uvue` 模板中的响应式对象属性可能出现类型推断不稳定；对布尔值、联合类型、可空值不要想当然地内联判断。

- Step 7. **目标平台编译验证**: 完成 Step 6 的首轮扫描后，必须尽快对用户要求的平台执行编译校验，直接使用 skill `verify-uniappx-compile-error`，并**必须并行**执行。
  **注意** 如果某一平台出现本机环境缺失 / IDE 配置缺失等问题，则在本次任务中无需重复调用相关能力，告知用户并直接跳过，除非用户明确指出手动修复了相关环境。
  **执行要求**:
  - 只有在前端示例页面已经真实 import 并调用目标插件 API 之后，才允许进入 Step 7。若当前仍停留在“仅插件实现、无页面真实调用”的阶段，必须停留在 Step 6 的逐文件 `lsp lint`，禁止提前执行 compile，并将结果视为插件有效性的结论。
  - 如果用户要求 Android、iOS、Harmony 等多平台支持，就必须分别验证这些平台，不能只验证单平台。
  - Step 6 与 Step 7 必须结合使用，不能机械地要求“所有 `lsp` 先清零，再开始 compile”；正确流程是先用 `lsp` 做首轮分流，再尽快用真实 compile 验证。
  - Step 6 与 Step 7 之间也必须保持串行关系：先完成首轮 `lsp lint` 分流，再执行真实 compile；禁止把 lint 与 compile 并发混跑，避免日志相互污染、误判阶段性成功信息或遗漏真实失败原因。
  - 如果 compile 已通过，而剩余 `lsp` 诊断仅集中在导包、混编符号解析、IDE 声明缺失等问题，应优先判定为 `lsp` / IDE 限制，保留已通过编译的实现，不得再为 `lsp` 强行改架构。
  - 如果 compile 失败，优先依据 compile 日志修复真实问题。
  - 必须区分“代码编译错误”和“本机环境缺失 / IDE 配置缺失”两类问题；前者必须修复，后者要明确告知用户缺少的环境项，例如 Xcode Command Line Tools、DevEco Studio 路径等。
  - 在至少完成目标平台的编译校验，并明确记录通过结果、`lsp`/compile 交叉验证结论或环境阻塞原因之前，禁止告知用户任务已完成。

- Step 8. **交付检查与总结**: 在 Step 7 完成后，必须执行最终交付检查，确认本次任务是否达到可交付状态，如果有缺失项需要补充。
  **检查项**:
  - 必须逐项核对用户要求支持的所有平台是否均已完成真实 compile 验证，并明确记录“通过 / 失败 / 环境阻塞”的结果。
  - 如果存在长时间阻塞且当前无法解决的问题，必须在最终回复中单独列出，并明确告知用户这些阻塞分别影响了哪些平台编译、下一步需要补齐什么环境或信息。
  - 只有在“各平台编译状态、阻塞项说明”全部输出清楚后，才允许告知用户任务完成或进入待确认状态。

  **总结项**
  - 过程中遇到的2次及以上语法校验报错和编译报错或者经过多轮修改才解决的问题需要总结到 AGENTS.md 中的常见代码问题章节。
