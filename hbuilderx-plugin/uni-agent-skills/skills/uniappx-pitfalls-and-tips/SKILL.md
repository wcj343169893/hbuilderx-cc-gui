### 参数说明

| 参数				| 类型	| 必填	| 说明																|
| ----------		| ------| ----	| ------------------------------------------------------			|
| `question`		| string| 是		| 问题描述，提问的问题												|
| `classification`	| string| 是		| 问题分类，取值范围：`uniapp`、`uni-app x`、`unicloud`。必须先按“分类选择规则”判断后再填写 |

### 执行逻辑

- Step 1. **提问** 参考`如何提问`章节，提出问题：

```shell
node {SKILL_BASE}/questionReply.js -q "{question}" -c "{classification}"
```

- Step 2. **整理回答**: 直接使用脚本返回的答复内容，结合用户上下文给出最终回答。
    * **如果命令调用失败**，需要回退到本地知识库查阅。

### 分类选择规则
- Step 1. **先识别问题主轴**：先判断用户核心是在问前端页面/组件/API，还是云端认证/数据/后端能力，不要因为问题里顺带提到 `uni-app x` 就固定选 `uni-app x`。
- Step 2. **按主轴选择分类**：
  - 选 `unicloud`：问题核心是 `uniCloud`、`uni-id`、云对象、云函数、云数据库、schema、后端鉴权、token、登录注册、权限控制、服务端接口设计。
  - 选 `uni-app x`：问题核心是 `uvue`、UTS、原生渲染、组件/页面实现、前端交互、样式布局、跨端渲染差异、仅 `uni-app x` 才有的 API/能力。
  - 选 `uniapp`：问题核心是传统 `uni-app` 页面、Vue 页面逻辑、普通组件/API 用法，且不以 `uniCloud` 或 `uni-app x` 专属能力为主。
- Step 3. **遇到混合问题时按主问题归类**：
  - 同时提到前端和 `uniCloud`，但重点在登录、注册、鉴权、云端数据结构、云函数/云对象选型时，优先选 `unicloud`。
  - 同时提到 `uni-app x` 和 `uniCloud`，但重点在页面如何写、组件如何实现、前端交互如何落地时，优先选 `uni-app x`。
  - 如果用户只是交代项目背景“这是一个 `uni-app x` 项目”，真正要问的是云端能力，仍然选 `unicloud`。

**【分类判断示例】**
- “`uni-app x` 项目接 `uni-id` 做账号密码登录，后端该选 `uni-id` 还是自定义云对象？” -> `unicloud`
- “`uni-app x` 登录页如何拆成独立页面，并在进入业务页前校验登录态？” -> `uni-app x`
- “普通 `uni-app` 页面里如何使用 `scroll-view` 实现下拉刷新？” -> `uniapp`

### 如何提问
**【核心调用原则：原子化提问】**
本技能更擅长处理单点技术，而非“大而全”的宏观业务需求。调用者禁止直接透传模糊、宏观的业务需求。在调用前，必须将用户需求拆解为底层 API、组件、原生能力等具体的原子化技术点。

**【提问学习示例（Few-Shot Learning）】**
* **❌ 错误示例：** “帮我写一个类似淘宝的电商 App。”（太泛，无法发挥知识库优势）
* **✅ 正确路径：**
  1.  “uni-app x 处理超长商品列表滚动时，如何结合 `list-view` 优化原生渲染性能？”
  2.  “插件市场是否有成熟的 UTS 原生扫码插件，支持在 uni-app x 渲染层直接覆盖 UI？”
  3.  “针对电商支付场景，如何处理 Android NDK 层的安全证书校验？”

* **❌ 错误示例：** “我的 App 崩溃了，帮我查一下。”（缺乏上下文）
* **✅ 正确路径：**
  1.  “检索 Issue 系统，查看 `uni-app x` 在 4.0 版本以后是否存在 `richtext` 内存溢出的已知 Bug？”
  2.  “在 UTS Plugin 插件开发时，如何打印日志到HBuilderX的运行控制台？”
