### 目标识别规则

- 优先从用户明确提到的文件路径、目录路径、页面名、组件名中识别目标范围。
- 若用户未提供路径，但当前编辑器存在活动文件，则优先将当前活动文件视为默认目标。
- 若用户明确提到“当前页面”“这个文件”，应结合编辑器上下文定位唯一目标文件。
- 若用户明确提到“这个目录下的页面”或“整个项目页面”，应分别识别为目录级改造或项目级改造。

### 范围判定规则

- 用户明确提供单个文件路径时，按单文件处理。
- 用户明确提供目录路径时，按指定目录处理。
- 用户明确要求“整个项目”“全部页面”“全量迁移”时，按项目级处理。
- 若用户描述无法唯一定位目标，应先发起一次简短追问，仅确认改造范围，不要提前展开实现。

### 追问规则

- 仅在以下场景允许追问：无法确定唯一目标文件或目录；用户同时提到了多个候选目标，但未说明优先级；用户未说明是改单页、目录还是整个项目，且当前上下文也无法推断。
- 追问内容应尽量简短，且只用于确认改造范围，例如：“要改当前文件、指定目录，还是整个项目页面？”

### 执行目标

- 将目标范围内的选项式 API 改造为组合式 API。
- 只做语法层面的转换，不主动调整业务逻辑、不改动原始行为。
- 保留所有已有代码与注释，避免遗漏、删减和语义漂移。
- 完成后确保无语法错误、无编译错误，并保持同一套代码在目标平台上的可编译性与跨端行为一致性。

### 触发条件

- 用户明确要求将选项式 API 改为组合式 API。
- 用户要求统一项目内页面或组件的脚本写法。
- 用户要求将旧示例、旧页面或旧组件迁移到 `<script setup>` 风格。
- 当前目标项目明确为 `uni-app x` 项目。

### 不适用场景

- 用户要求同时重构业务逻辑、状态流、接口调用方式或组件拆分方式。
- 用户目标是迁移到其他技术体系，而不是在 uni-app x 内做组合式改造。
- 原文件已经是组合式 API，且不存在明确的兼容性或语法问题。
- 当前目标项目是传统 `uni-app`（非 `uni-app x`）项目。

### 前置检查

- 确认目标文件确实使用选项式 API，而不是混合了少量组合式语法的过渡态文件。
- 确认用户是否限制改造范围；若范围不明确，优先按当前页面或指定文件处理，不默认扩散到全项目。
- 若涉及多个平台，优先保持跨端一致行为，不为单一平台单独引入业务改写。
- 确认项目文件形态与 `uni-app x` 一致，优先面向 `.uvue`、`.uts` 等文件执行改造规则。

### 执行逻辑

- Step 1. **确认改造范围**：根据用户描述、当前活动文件和编辑器上下文自动确定待改造页面或组件列表。

- Step 2. **识别可改造文件**：优先处理 `uni-app x` 项目中的 `.uvue` 页面与组件文件；`.uts` 仅在被 `.uvue` 迁移联动影响、需要同步修正类型、辅助逻辑或导出结构时才调整，避免误改传统 `uni-app` 的 `.vue` 页面或无关文件。

- Step 3. **执行语法转换**：将选项式 API 按组合式 API 规则改写，遵循以下约束：
  - 只做语法转换，避免引入新的业务逻辑分支。
  - 不删除、不省略任何已有代码和注释。
  - 在 `uni-app x` 组合式 API 中，所有 Vue 公开 API 都不需要手动 `import`，`uni-app x` 会自动引入。
  - 同时，页面生命周期和组件生命周期等常用组合式 API 也应直接使用，不额外书写手动 import。
  - 若改造目标本身已存在这类手动 import，改造后应一并移除，避免输出冗余或与 `uni-app x` 项目约定不一致的代码。
  - 错误示例：`import { ref, computed, watch, getCurrentInstance, onMounted } from 'vue'`、`import { onLoad, onShow } from '@dcloudio/uni-app'`。
  - 正确要求：直接在 `<script setup>` 中使用 `ref`、`reactive`、`computed`、`watch`、`watchEffect`、`getCurrentInstance`、`onLoad`、`onPageShow`、`onMounted` 等 API，不额外书写手动 import。
  - 选项式 API 中依赖 `this` 的访问，在组合式 API / `<script setup>` 中不能继续使用 `this`；需要改为通过getCurrentInstance()!.proxy!获取实例代理后再访问对应成员，参考文档{knowledges_base_dir}/uni-app-x/docs/vue/composition-api.md#getCurrentInstance。
  - 选项式 API 中的 `onShow`、`onHide`，改造后分别使用 `onPageShow`、`onPageHide`。
  - 组件生命周期兼容时：
    - `beforeUnmount` 改为 `onBeforeUnmount`
    - `mounted` 改为 `onMounted`
    - `unmounted` 改为 `onUnmounted`
    - `created` 改为 `<script setup>` 顶层执行逻辑

- Step 4. **执行声明顺序自检**：完成 `<script setup>` 改写后，必须对顶层声明做一次依赖顺序检查：
  - 强制规则：在 `uni-app x` 的 `<script setup lang="uts">` 中，所有变量、函数、`computed`、`watch`、生命周期回调及 `defineExpose` 暴露成员，必须遵循“先声明、后使用”；禁止任何前向引用，即只要某段逻辑中引用了某个标识符，该标识符就必须先出现在源码中。
  - 函数依赖处理：若函数 A 调用函数 B，或回调 / 闭包中调用函数 B，而 B 定义在后面，必须按依赖关系重排顺序，优先将“底层被调用函数”放前，将“事件入口函数”放后。
  - 函数表达式禁用场景：`const fn = () => {}` 不得用于递归调用、回调中的自引用、或在闭包 / Promise / 弹窗回调中再次引用自身；这类场景必须改为函数声明（如 `function fn() {}`），并且函数定义位置仍需放在所有调用点之前。
  - 高风险场景：事件处理函数、表单校验函数、`computed` / `watch` / 生命周期回调、`defineExpose` 暴露成员，以及匿名函数、弹窗回调、Promise 回调中的前向引用；递归调用、互相调用链也必须重点检查。
  - 推荐输出顺序：`import` -> `type` -> 常量 / `ref` / `reactive` -> `computed` -> 纯工具函数 -> 被复用的底层业务函数 -> 事件处理函数 -> 生命周期 -> `defineExpose`。
  - 当用户要求最终通过编译时，若编译报错来自其他已迁移为 `<script setup lang="uts">` 的同类页面，允许做同类最小修复，以消除同模式的声明顺序问题。

- Step 5. **处理 `ref` 类型**：在组合式 API 中，严格按引用目标声明 `ref` 类型：
  - 内置组件（如 `scroll-view`、`swiper`、`slider`）：使用 `Uni组件名(驼峰)Element | null`，例如 `UniSliderElement | null`。
  - 内置组件类型声明错误示例：在 `swiper` 上写成 `const swiper = ref<UniElement | null>(null)` 属于错误写法，应改为对应的 `UniSwiperElement | null`。
  - 内置元素（如 `view`、`text`）：使用 `UniElement | null` 或 `UniElement[] | null`。
  - 自定义组件：使用 `ComponentPublicInstance | null` 或 `ComponentPublicInstance[] | null`。
  - 每个 `ref` 必须单独声明，变量名与模板中的 `ref` 名称保持一致。
  - 当页面或组件中已存在同名业务字段（如 `username`、`password`）时，模板 `ref` 命名应优先使用 `xxxRef` 形式（如 `usernameRef`、`passwordRef`），避免与表单值或业务变量重名；除非原模板已有稳定约定，否则不要使用语义较弱的 `xxxInput`、`xxxEl` 作为默认替代。
  - 访问 `ref` 时直接使用 `xxx.value`。
  - 避免 `ref` 变量名与 import 的组件名重复。
  - 除非为消除命名冲突、类型错误或父子引用断链风险，避免无故修改模板中的 `ref` 名称。

- Step 6. **处理父子组件暴露关系**：
  - 组合式 API 下，子组件方法不会自动暴露给父组件。
  - 父组件需要通过 `ref` 调用的方法，必须在子组件中通过 `defineExpose` 显式暴露。
  - 测试页或外部调用方若依赖页面数据或方法，也必须通过 `defineExpose` 显式暴露。
  - `defineExpose` 不是默认迁移动作，不能为了“保险”导出整个页面或组件作用域；只有在确认存在父组件、页面调用链、测试文件（如 `test.js`）或其他外部使用方通过 `ref` 访问该页面/组件实例成员时，才按最小范围暴露必要的数据和方法。
  - 如果当前目标文件内看不到明确的外部调用证据，也未在用户指定范围内发现对应引用关系，则禁止主动添加 `defineExpose`。

- Step 7. **处理组件迁移专属事项**：若目标是组件而非页面，除 `data`、`computed`、`methods`、生命周期迁移外，还必须同步检查并迁移 `props`、事件声明、`defineExpose`、模板 `ref` 调用链，以及父组件对该组件的引用关系。

- Step 8. **处理页面生命周期特例**：`onPullDownRefresh` 必须按页面生命周期函数处理，不能改造成普通方法。

- Step 9. **进行语法与编译校验**：
  - 若修改了 `.uvue` 或 `.uts` 文件，必须调用 `uniappx-syntax-checker` 进行语法校验。
  - 完成改造后，必须调用 `verify-uniappx-compile-error` 检查编译错误。
  - 必须检查控制台中的编译报错或编译日志中的错误输出；若发现报错，应自行修复后再次检查。
  - 如果存在报错，需要继续修复并重复校验，直到无报错且编译通过，或达到终止条件。

### 结果返回要求

- **If Success**：返回已修改文件列表、选项式生命周期到组合式生命周期的关键映射情况、是否新增 `defineExpose`、`ref` 类型修正、声明顺序调整情况，以及语法校验和编译校验结果。
- **If Failure**：返回失败文件、当前报错信息、已尝试修复的步骤，以及阻塞继续自动处理的具体原因；如果存在历史遗留问题，需区分“本次新增问题”和“原有问题”。

### 执行指导原则

- 优先做最小改动，避免无关重构。
- 如果用户只要求改单页或单文件，禁止扩散到整个项目。
- 如果原文件中存在混合写法、宏条件编译或复杂父子调用链，先保持行为一致，再处理组合式重组。
- 如果连续修复 5 次仍无法通过语法或编译校验，应停止并请求人工介入。
- 本 skill 仅用于 `uni-app x` 项目，不用于传统 `uni-app` 的 `.vue` 页面迁移。

### 失败回退策略

- 如果在转换过程中发现原文件依赖隐式 `this` 上下文且难以安全平移，应先保留原始结构并仅转换可确定部分。
- 如果父子组件调用依赖未显式声明的方法暴露关系，应先补齐 `defineExpose`，再继续验证调用链。
- 如果编译错误来自历史遗留问题且并非本次改造引入，应在结果中明确区分“新增问题”和“原有问题”。
- 如果目标目录存在大量关联文件且用户未授权批量改造，应停止扩散修改并要求用户明确范围。
